Attribute Enrichment for Ecommerce Discovery

Self-improving attribute enrichment system powered by ML and behavioral feedback loops

Role Solo — PM + engineer
Year 2025
Status Working prototype
Stack React · Hooks · Zero external dependencies

Problem

Ecommerce discovery depends on structured product attributes — color, size, material, style, occasion, brand tier. Most catalogs have incomplete and inconsistent data. That translates directly into poor search relevance, weak filtering, high zero-result rates, and lost revenue.

The root issues:

  • Long-tail SKUs have no attributes. High-velocity items get manually enriched. Everything else is missing data that never gets touched.
  • No confidence signal. Existing systems treat all attribute values as equal — a value manually verified by a merchant and one scraped from a product title are indistinguishable.
  • No feedback loop. Behavioral signals — what users search for, filter on, and convert from — aren't fed back into the attribute data. The system doesn't learn.

Product vision

Build a self-improving attribute enrichment system powered by ML and behavioral feedback loops that continuously enhances data quality and drives measurable conversion lift.

Attribute enrichment should evolve from a data cleanup task into a core revenue-driving system that learns from customer behavior. The north star metric is revenue lift — driven by improvements in conversion rate, search success rate, and facet engagement.

System overview

LayerInputsOutputs
Ingestion Product catalog, behavioral data (searches, clicks, conversions), embeddings Normalized product records with attribute candidates
Processing Product titles, descriptions, similar-item embeddings, behavioral signals Attribute values with confidence scores and source labels (catalog / ml_title / ml_embedding / behavioral)
Output Enriched attribute store Improved filter facets, better search ranking, reduced zero-result rate
Feedback loop Conversion signals, facet engagement, search refinements Confidence score updates — system improves continuously without manual re-enrichment

What the prototype does

Single-file React component. 12 SKUs across 6 categories. 6 tracked attributes per SKU. Four interactive tabs:

Catalog tab — confidence-aware attribute view

Every attribute cell shows its value, confidence score (color-coded green/yellow/red), and source label. Filter by All / Missing / Low confidence / Needs review. Click "Run Enrichment" to step the ML pipeline through each SKU — missing attributes fill in with animated highlights.

Review Queue tab — human-in-the-loop

Medium-confidence attributes (65–85%) surface for human review. Accept promotes to verified and bumps confidence. Reject clears the value and flags it for re-enrichment. Every approval action updates the live KPIs.

Signals tab — behavioral feedback loop

"Simulate Session" generates realistic behavioral events — searches, filter clicks, add-to-carts, conversions. Each event targets a specific attribute on a random SKU and boosts its confidence score. The feedback loop is visible in real time: watch a low-confidence "occasion" value on a yoga pant climb after workout-related searches.

Facet Preview tab — before vs. after

Side-by-side comparison of filter coverage and estimated zero-result rate before and after enrichment. Shows exactly what running the pipeline unlocks for customers — more filter options, fewer dead ends.

KPI framework

  • North star: Revenue lift. Enriched attributes improve search relevance and filter coverage — both drive conversion.
  • Attribute coverage. Filled / total possible attributes across the catalog. Tracks enrichment progress.
  • High-confidence accuracy. % of filled attributes at ≥85% confidence. Quality signal, not just quantity.
  • Pending review queue size. Medium-confidence attributes awaiting human verification. A leading indicator of data quality lag.
  • Zero-result rate. Estimated % of filter combinations that return no products. Direct proxy for discovery failure.
  • Facet engagement. % of sessions that use at least one filter. Enriched attributes expand filter options — more engagement means the data is working.

Key design decisions

Confidence as a first-class data field

Every attribute carries a confidence score and a source label. A catalog-sourced value at 0.97 and an embedding-inferred value at 0.71 are treated differently — in the UI, in the review queue, and in the feedback loop. Confidence isn't metadata; it's part of the data model.

Human-in-the-loop for medium confidence

Attributes between 65–85% confidence don't auto-publish. They surface in a review queue where a merchant or data team can accept or reject. This threshold is configurable. High-confidence values (≥85%) publish automatically. Low-confidence (<65%) go back for re-enrichment.

Behavioral signals as a confidence update mechanism

When a user searches "merino sweater" and clicks a product, that's a signal that the material attribute is accurate. When a "slim fit" filter surfaces a purchase, style confidence increases. The feedback loop closes the gap between what the catalog says and what customers confirm with their behavior.

Source attribution on every attribute

Each attribute value is labeled with its origin: catalog, ml_title, ml_description, ml_embedding, or behavioral. This matters for debugging, auditing, and understanding which enrichment sources are performing. A high-confidence ml_embedding value that keeps getting rejected in the queue is a signal the embedding model is failing on that attribute type.

Learnings

  • Confidence scoring changes everything downstream. Once every attribute has a confidence score, prioritization becomes obvious — review queues sort by confidence, facets can suppress low-confidence values, and the feedback loop has a clear signal to optimize.
  • The feedback loop is the defensible moat. ML enrichment is table stakes — any vendor can run attribute inference. A system that gets smarter from your specific customers' behavior is not replicable.
  • Zero-result rate is the most visceral KPI. It's easy to talk about "attribute coverage." It's harder to ignore "23% of filter combinations on this category return zero products."
  • Human review should be scoped narrowly. Routing everything to a human defeats the purpose. The right scope is medium confidence only — and even that queue should shrink as the feedback loop matures.
