Royals Player Insights Dashboard

Sports analytics engine with forecasting, causal inference, and anomaly detection

Role Solo — data PM + analyst
Year 2025
Status Complete
Stack Python · pandas · numpy · statsmodels · DoWhy · EconML · pybaseball · matplotlib

Problem

MLB player performance data is public but scattered. Standard tools give you box scores and basic stats. Nobody gives you a single local tool that does forecasting, feature importance, causal inference, and directionality analysis on a player you care about — without paying for a vendor platform.

The goal: build a real analytics engine for KC Royals players. Something that goes past "what happened" and into "what drives performance" and "what will happen next." The gap isn't data availability — it's analytical depth, accessible without a data team.

What the engine does

ModuleMethodOutput
Forecasting SARIMAX time-series model. Falls back to EWMA when statsmodels is unavailable. Projected player metrics over a date range. Saved to ./outputs/.
Feature Importance Identifies inputs that drive a target metric for a given player and season range. Ranked feature list with relative importance scores.
Directionality (PDP) Partial dependence plots — how each feature influences outcome direction and shape. PDP charts per feature, saved to ./outputs/.
Causal Inference DoWhy or EconML (soft dependency). Estimates a causal effect, not just correlation. Causal estimate with confidence bounds.
Anomaly Detection Flags outlier games or stretches in the performance data. Labeled anomaly periods with deviation metrics.

Usage

One command. No UI required.

python app.py --mode cli --player "Bobby Witt Jr." --start-year 2023 --end-year 2025 python app.py --run-tests

All outputs — forecasts, feature importance plots, PDP charts, anomaly flags — save to ./outputs/. Streamlit UI is available as a lazy optional import, not a dependency.

Key decisions

CLI-first, UI as an option

Streamlit was removed from the core dependency stack. The engine runs in full from a terminal command — automatable, scriptable, CI-friendly. UI is additive. For data tools, the CLI is the real interface.

Synthetic data fallback

Realistic demo data is generated procedurally when pybaseball is unavailable or rate-limited. The tool works offline, in CI, and in sandboxes without live data. This was a decision to prioritize portability over raw data fidelity in the development loop.

Soft dependency stack

statsmodels, DoWhy, EconML — all optional. When they're missing, the engine degrades gracefully to simpler methods. Hard requirements: pandas and numpy only. This makes installation lightweight and the tool approachable for users who don't need every capability.

Causal inference as a first-class module

Most sports analytics tools show correlation. Including a causal estimate — even a simplified one — frames a different question: not "what correlated with Bobby Witt Jr.'s strikeout rate" but "what caused it." That's a different conversation, and it's where analytical depth shows.

What this demonstrates

  • Data PM depth. Not just pulling stats — building a forecasting and causal analysis pipeline from scratch, understanding which methods apply and why.
  • Python fluency. pandas, numpy, statsmodels, matplotlib, DoWhy, EconML — working across the full analytics stack.
  • System design for real-world constraints. Fallback logic, graceful degradation, offline mode — built for environments where things go wrong.
  • Going past correlation. Feature importance shows what matters. PDP shows direction. Causal inference shows effect. That's a complete analytical picture, not just a dashboard.

Learnings

  • Graceful degradation is underrated. Writing fallback logic — synthetic data, EWMA, no-Streamlit mode — took extra time upfront but made the tool actually usable in every environment. Systems that fail hard on missing dependencies don't get used.
  • Causal inference is a differentiator in analytics. Most tools show correlation. Showing a causal estimate changes the conversation from "interesting" to "actionable."
  • CLI-first is the right default for data tools that need to run in automation or CI. UI is additive, not the foundation. Design for the terminal and the UI is easy to add later; design for the UI and the terminal is an afterthought.
  • Soft dependencies are a product decision. Choosing which capabilities require which packages shapes who can use the tool and where. Keeping the hard requirement surface small made adoption easier.
