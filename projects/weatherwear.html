WeatherWear

Personalized clothing recommendation engine

Role Solo — PM + engineer
Year 2025
Status Complete prototype
Stack React · JSX · Lucide · Zero external dependencies

Problem

Weather apps tell you the temperature. None of them tell you what to wear. And everyone's body is different — 45°F feels like two entirely different things to someone who runs cold vs. someone who runs hot, or a 65-year-old vs. a 28-year-old.

No tool personalizes clothing recommendations to the actual person, not just the weather. This is a classic personalization gap: the input (weather) is generic, but the need (what to wear) is individual.

How the engine works

Six environmental signals are extracted from weather input and normalized to a 0–1 scale. A user profile adjusts every signal. Dynamic weights amplify active signals and dampen inactive ones. A scoring function ranks 67 clothing items. Everything is explainable.

SignalWhat it capturesProfile adjustments
Cold Stress Cold protection need. Peaks below -20°F, zero above 65°F. Thermal tendency, age cold sensitivity, activity warmth
Heat Stress Cooling need. Zero below 70°F, peaks above 110°F. Thermal tendency, age heat sensitivity, activity warmth
Wind Exposure Wind chill risk on a 0–50 mph scale, amplified by cold. Indirect via cold stress interaction
Precipitation Snow = 1.0, heavy rain = 0.85, light rain = 0.5. Environmental only
UV Exposure Sun protection on 0–11 UV index scale. Environmental only
Humidity Discomfort Muggy conditions. Only activates when hot. Indirect via heat stress

User profile — three dimensions:

  • Thermal tendency — runs cold / neutral / runs hot. Applies a ±5°F offset to effective temperature.
  • Age bracket — under 30 through over 65. Increases cold and heat sensitivity with age.
  • Activity level — sedentary / moderate / very active. Active users generate body heat, reducing cold stress.

Scoring: weighted dot product of signal × weight × item_relevance across all 67 items (35 everyday, 32 athletic). Items below 0.12 threshold are excluded. Results labeled Essential / Recommended / Optional.

The weighting system

Weights are not static. Each feature's weight is amplified when its signal is active and dampened — but never zeroed — when inactive. Dampening without zeroing prevents blind spots: on a mild but slightly windy day, wind still registers.

A 30% bias cap prevents any single signal from dominating. Without it, a blizzard collapses all recommendations to cold-weather gear only. Excess weight above the cap redistributes proportionally to uncapped features via an iterative loop. All weights normalize to sum to 1.0.

This is the core engineering problem. Getting the redistribution logic right — bias cap plus iterative normalization — matters more than the item database. A better algorithm on the same items beats a bigger item catalog with a worse algorithm.

Explainability

Every recommendation shows its numeric score, a color-coded feature contribution bar, and the top driving factor. You don't just get "wear a rain jacket" — you get why: Precipitation 0.85, driving 34% of this recommendation's score.

The Feature Importance panel shows all six signal strengths and their capped weights in real time. As you change weather inputs, weights shift live. That transparency is a design choice: the system earns trust by showing its work.

Feedback loop

Each recommendation is ratable: accurate (✓), too warm (🔥), too cold (🥶), or irrelevant (✗). Ratings drive live metrics displayed in the UI:

  • Accuracy — % of rated items marked accurate
  • Precision — true positives / (true positives + false positives)
  • Recall — true positives / (true positives + false negatives)
  • F1 Score — harmonic mean of precision and recall
  • Coverage — % of recommendations that have been rated

The full session — weather input, profile, extracted features, weights, scored recommendations, feedback labels, and computed metrics — exports as structured JSON. That's a training record. The feedback loop was designed for model iteration from the first build.

Key decisions

No API key required

Manual sliders plus 8 US city presets. Portable, demo-ready, works offline. Removing the backend dependency was a scope decision: prove the algorithm first, infrastructure later.

Single file

The entire engine — feature extraction, weighting, scoring, UI, feedback loop — is one weatherwear.jsx file. No build config, no external dependencies beyond React and Lucide. Drop it into any React project and it runs.

Metrics panel built first

Precision/recall/F1 were wired in before the item database was finalized. Building the metrics layer upfront forced clarity on what "accurate" means before the system was complete. Most teams add metrics as an afterthought. Here they shaped the design.

Learnings

  • Dynamic weights matter more than the item catalog. The redistribution algorithm — bias cap plus iterative normalization — is the core IP. Swapping items is easy. Getting the weights right is the hard part.
  • Effective temperature beats raw temperature for personalization. Profile adjustments are small (±5°F) but compound across multiple features, producing meaningfully different recommendations for different users in the same weather.
  • Explainability is not a feature — it's a trust mechanism. Users who see why an item was recommended are more likely to rate it accurately, which improves training data quality.
  • Building the feedback loop first changed the design. Knowing the system would produce training data shaped what signals were tracked, how items were labeled, and what the export format looked like.
