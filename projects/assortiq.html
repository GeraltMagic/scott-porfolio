AssortIQ

AI-powered assortment optimization for mid-market retail

Role Solo — PM + engineer
Year 2025
Status Complete
Stack Python · FastMCP · Anthropic SDK · Chronos · sentence-transformers

Problem

Retail buyers and category managers make high-stakes assortment decisions — what to carry, how deep, where — using fragmented tools and gut instinct. The workflow: pull sales data from one system, check inventory in another, open a spreadsheet, manually cross-reference vendor terms, make a call. For one store format, that's painful. Across multiple banners, channels, and geographies, it breaks.

Three forces compound the problem:

  • Spreadsheet dependency. Most mid-market retailers plan assortments in Excel. No version control, no collaboration, no real-time demand signal. Errors compound silently.
  • Localization is a guess. Most retailers apply a national assortment with manual overrides per cluster. That's exception management, not optimization.
  • Existing tools are enterprise-only. Blue Yonder, RELEX, and Oracle cost seven figures and require 6–18 months to implement. No product serves the mid-market (50–2,000 locations) who needs AI-driven decisions without the enterprise overhead.

Who it's for

Primary — Category Manager. Owns a category P&L. Decides what's in and what's out. Drowning in spreadsheets. Can't see cross-store performance in real time. Relies on vendor sell sheets instead of demand data. Has budget authority for tools.

Secondary — Buyer / Merchant. Executes the buy. Negotiates with vendors. Buys blind — doesn't know what's actually selling at store level until it's too late. Overbuys safe SKUs, underbuys emerging ones.

Executive stakeholder — VP of Merchandising. Sets assortment strategy across categories. Can't answer "are we over-indexed on private label in the Southeast?" without a week of analyst time.

V1 targets the Category Manager. They own the decision, feel the pain daily, and control the budget.

What I built

An MCP server with 6 tools and 3 resources, built on FastMCP and connected to Claude via the Anthropic SDK. The model drives orchestration — it decides when and how to call the tools based on what the user is asking. Two working prototypes — a live Gradio Space and an enhanced terminal REPL — call the same tool functions, proving the architecture end-to-end.

ToolWhat it doesUsed in
forecast_demand SKU-level demand forecasting using Chronos-Bolt time-series model. Falls back to EWMA when the pipeline isn't available. Copilot, Simulation
find_similar_items Embedding-based SKU similarity using BGE sentence-transformers. Solves the "what's like this?" question at scale. Copilot
cluster_stores Auto-segments stores into data-driven clusters based on sales patterns and format — not manual tiers. Business View
get_assortment_recommendations Add/drop/swap recommendations at the store-cluster level, with rationale. Human approves or overrides every call. Business View, Simulation
get_category_health Scorecard per category: revenue, margin, sell-through, out-of-stocks, SKU redundancy (cannibalizing items). Business View, Copilot
search_skus Semantic search across the SKU catalog. Natural language queries return ranked matches. Copilot

Resources available to the model at inference time: 100-SKU catalog, 20-store attribute set, and aggregate sales summary.

Working prototypes

Two surfaces that call the same underlying tool functions as the MCP server — proving the architecture works end-to-end with real ML models.

Gradio Space — Try it live →

Three-tab web app deployed on HuggingFace Spaces. Business Overview tab loads category health scores and store clusters on startup. Copilot tab accepts natural language queries and routes them to real tool calls (search, similarity, forecast, health, recommendations). Simulation tab runs add/drop/swap recommendations with forecast impact projections. Uses lighter models for the free tier: Chronos-Bolt-Tiny for forecasting, all-MiniLM-L6-v2 for embeddings.

Terminal REPL — MerchAssistant

Enhanced Anthropic SDK-powered chat assistant with slash commands that bypass the API for instant results. /overview renders a Rich-formatted category health table across all five categories. /diagnose <category> shows health panels, at-risk SKUs, and similar items. /simulate <category> displays add/drop/swap recommendations with forecast comparison. All commands call the same Python functions as the MCP server and format output with Rich tables and panels.

Key decisions

MCP over a REST API

Building as an MCP server means the tools are natively callable by Claude. No custom client code, no routing logic — the model handles orchestration. The PM and engineer roles collapse into one faster feedback loop.

Chronos for zero-shot forecasting

Chronos-Bolt works without training data. Cold-start SKUs — new items with no sales history — get a forecast out of the box. Traditional methods fail here. This is a hard problem for the category manager; it's solved at the infrastructure layer.

Phased rollout: V1a then V1b

Full scope is 7 capabilities across 3 external data integration layers (Circana, Nielsen, competitor data). V1a ships scorecard, recommendations, clustering, and one syndicated source — proving value without betting on all integrations at once. V1b adds forecasting, experimentation, and new item introduction.

Human approval on every recommendation

"Drop SKU X in the Southeast cluster. Replace with SKU Y." The system recommends. A human approves or overrides. AI as decision support, not autopilot — category managers have context the model doesn't.

Positioning against enterprise incumbents

Blue Yonder and RELEX sell to Walmart. The pitch here is speed-to-value: deploy in weeks, not months. No dedicated planning team. SaaS pricing that doesn't require a board-level procurement decision. Toolio is the closest mid-market comp but focused on financial planning, not assortment intelligence.

Data architecture

  • Internal data layer. POS, inventory, planogram, loyalty. API connectors to Oracle Retail, SAP, Shopify, custom EDI.
  • Syndicated data layer. Phase 1: Circana integration — category trends, new item velocity, market share shifts. Phase 2: Nielsen added.
  • Competitive data layer. Phase 1: third-party aggregator (Competera or similar). Phase 2: direct web scraping with legal review.
  • Vendor data layer. Structured upload with optional EDI/API from major CPGs.

Explicitly out of scope for V1

Space/planogram optimization (different product). Vendor negotiation or procurement workflow. Price optimization. Full supply chain or replenishment. Scope discipline is a product decision — every "not now" here is intentional.

Learnings

  • Zero-shot forecasting is a legitimate unlock for cold-start problems. Writing this into the architecture from day one — rather than bolting it on later — is a product decision, not just a technical one.
  • MCP dramatically simplifies orchestration. The model calls the right tools in the right order. Less glue code, more capability, faster iteration.
  • Phasing external data integration was the right call. Nielsen and Circana APIs are gated and expensive. Prove the value of external data with one source before investing in broader coverage.
  • The hardest part of this product is not the AI — it's change management. Category managers have their own systems and instincts. The recommendation engine has to be trustworthy before it's adopted.

Success metrics

  • Assortment review cycle time: weeks → days
  • Sell-through rate lift for optimized categories
  • SKU count reduction without revenue loss
  • New item success rate — % meeting velocity targets within 90 days
  • Experiment-to-rollout conversion rate
