AssortIQ

AI-powered assortment optimization for mid-market retail

Role Solo builder (PM + engineer)
Year 2025
Status Complete
Stack Python · FastMCP · Anthropic SDK · Chronos · sentence-transformers

Problem

Retail buyers and category managers make high-stakes assortment decisions — what to carry, how deep, where — using fragmented tools and gut instinct. Pull sales data from one system, check inventory in another, open a spreadsheet, manually cross-reference vendor terms, make a call. For one store format, that's painful. Across multiple banners and geographies, it breaks.

Three forces make this worse:

  • Spreadsheet dependency. Most mid-market retailers still plan assortments in Excel. No version control, no collaboration, no real-time demand signal.
  • Localization is a guess. Most retailers apply a single national assortment with manual overrides. That's exception management, not optimization.
  • Existing tools are built for enterprise. Blue Yonder, RELEX, and Oracle cost seven figures and take 6–18 months to implement. No product serves the mid-market retailer who needs AI-driven decisions without enterprise overhead.

Target Users

Primary — Category Manager. Owns a category P&L. Decides what's in and out. Drowning in spreadsheets. Can't see cross-store performance in real time.

Secondary — Buyer / Merchant. Executes the buy. Buys blind on what's selling at store level until it's too late. Overbuys safe SKUs, underbuys emerging ones.

V1 is built for the Category Manager — they own the decision, feel the pain daily, and have budget authority.

Solution

MCP server with 6 tools and 3 resources, built on FastMCP and integrated with Claude via the Anthropic SDK.

Tool What it does
forecast_demand SKU-level demand forecasting using Chronos-Bolt time-series model with EWMA fallback
find_similar_items Embedding-based SKU similarity using BGE sentence-transformers
cluster_stores Auto-segments stores into data-driven clusters based on sales patterns and format
get_assortment_recommendations Add/drop/swap recommendations at the store-cluster level
get_category_health Scorecard per category: revenue, margin, sell-through, out-of-stocks, SKU redundancy
search_skus Semantic search across the SKU catalog

Resources: 100-SKU catalog, 20-store attributes, and aggregate sales summary — all available to the model at inference time.

Key Decisions

MCP over REST API

Built as an MCP server so the tools are natively callable by Claude. No custom client code needed — the model drives the orchestration.

Chronos for forecasting

Zero-shot time-series model that works without training data. Handles cold-start SKUs out of the box. Falls back to EWMA when the pipeline isn't available.

Phased rollout (V1a / V1b)

Full scope spans 7 capabilities with 3 external data integration layers. V1a ships scorecard, recommendations, clustering, and one syndicated source. V1b adds forecasting, experimentation, and new item introduction. Keeps time-to-value tight without scoping down ambition.

Human approves every recommendation

The system recommends — add SKU X, drop SKU Y, run a test in the Southeast cluster. A human approves or overrides. AI as decision support, not autopilot.

Competitive Positioning

The incumbents (Blue Yonder, RELEX, Oracle) sell to Walmart and Tesco. Position against them on speed-to-value: deploy in weeks not months, no dedicated planning team, SaaS pricing that doesn't require a board-level procurement decision.

Toolio is the closest mid-market comp but stronger on financial planning than assortment intelligence. The differentiator is the recommendation engine — not "here's your data organized" but "here's what to do about it," informed by demand signals, store clusters, and competitor intelligence.

Learnings

  • Zero-shot forecasting is a legitimate unlock for cold-start problems. Chronos handles new SKUs without historical data — that's a hard problem for traditional methods.
  • MCP dramatically simplifies tool orchestration. Writing tools as MCP resources means the model handles when and how to call them. Less glue code, more capability.
  • Phasing on external data integration was the right call. Nielsen and Circana APIs are gated and expensive. Prove value with one source before investing in broader integration.
